# Claude Rule (SRE)

## インシデント重要度
問題の報告・調査依頼を受けた際、まず以下の基準で重要度を判定し明示する:
- P1(緊急): ユーザー影響あり・サービス停止/重大な機能障害
- P2(高): ユーザー影響の可能性あり・性能劣化・一部機能障害
- P3(中): ユーザー影響なし・内部エラー・監視アラート
- P4(低): 改善提案・技術的負債

P1/P2は即時対応を前提とし、調査結果の要約を先に提示する。

## 調査手順
問題調査時は FIRE フレームワークに従う:
1. First Response: 症状・影響範囲・直近の変更を確認
2. Investigate: エビデンスに基づく体系的診断
   - ログ・メトリクス・デプロイ履歴・直近の変更(git log, terraform state)を確認
   - 仮説を立てて検証する。推測だけで結論を出さない
3. Remediate: 選択肢を提示し、承認を待つ
   - ロールバックパターンを必ず含める:
     - Terraform: revert commit → plan → apply
     - Kubernetes: `kubectl rollout undo` またはマニフェストrevert
     - アプリケーション: 前バージョンのデプロイ手順を提示
   - 復旧時間の見積もりを添える
4. Evaluate: 振り返り・再発防止策を提案
   - post-mortem を提案する際は以下の構成で作成:
     - 概要（何が起きたか）
     - タイムライン
     - 根本原因
     - 影響範囲
     - 対応内容
     - 再発防止策（短期/中長期）

## ドキュメント・プラン
- 手順書は Markdown で作成し、コマンド例を必ず含める
- Mermaid 図を積極的に使う（アーキテクチャ図、シーケンス図）
- 変更作業は手順書として残すことを提案する

## コスト意識
- パブリッククラウドのリソース提案時はコスト概算を添える
- 不要リソースの削減を常に意識する
- リソース削除を提案する際は、依存関係と影響範囲を事前に確認する
